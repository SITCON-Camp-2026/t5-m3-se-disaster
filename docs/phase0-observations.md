# Phase 0 Observations

## 我們做出了什麼？

- 首頁有原始資訊、整理工作台、訊息分類三個區塊。
- 訊息分類頁會用保守文字規則先分出服務類型，例如清泥 / 挖土、雨鞋 / 物資、醫療 / 藥、食物 / 飲水、交通 / 接送。
- 同一筆原始資訊可以有多個服務輸出，不再被強迫只放進一個分類。
- 分類頁現在把派工準備度、原始查核狀態與「派工前要先確認」放在同一張卡片上，避免把分類草稿看成已確認資料。

## 哪些資料讓 agent 很容易整理？

- 原文含有明確需求詞的資料比較容易先分到候選服務類型，例如清泥、雨鞋、藥品、飲用水、道路封閉。
- 這些分類只能代表文字線索，不代表需求已確認，也不代表可以派工。

## 哪些資料讓 agent 亂補或誤判？

- 只看到「水」字時容易被誤放進食物 / 飲水；因此分類規則必須避免把水電檢修當成飲水需求。
- 同一筆資料可能同時包含物資、水電、集合點或道路狀態；如果 UI 只允許單一分類，會把其他重要線索藏起來。
- 程式若替每筆資料補上概略地點，會讓畫面看起來像已整理出現場位置；這次已移除這種補值。
- 「官方公告」來源類型不等於已查核，仍要看原文是否說明日期、來源與同步狀態。

## 哪些資訊不能直接變成任務？

- 缺少明確地址、時間、同意狀態或現場條件的資訊不能直接變成任務。
- 由群組、留言、截圖、家屬或第三方轉述而來的資訊，需要先確認來源與當事人狀態。
- 原文提到「不知道」「不確定」「疑似」「尚未」「無法確認」時，應先保留為待確認資訊。

## 哪些資訊的操作者不是當事人？

- 志工代長者轉述的資訊，需要確認當事人同意與完整需求。
- 外地家屬來電的資訊，需要確認親友現況與位置，不能只靠來電建立任務。
- 社群貼文、群組留言、截圖都可能不是第一手資訊。

## 我們剛剛偷偷假設了什麼？

- 曾經假設分類完成就可以顯示為「已分類」，但這會讓人誤以為 agent 的分類已可信。
- 曾經用固定規則補概略地點，這是原文沒有明確給出的資訊。
- 服務類型只是整理草稿，不等於後續流程真的需要派哪一種志工。

## 如果要讓這個工具變可靠，下一步需要什麼？

- 需要可編輯的整理草稿，讓人類記錄採用或修正 agent 判斷的理由。
- 需要更清楚的派工前欄位，例如地點可用性、時間有效性、同意狀態、來源是否第一手。
- 需要把「分類草稿」和「已確認可行動」拆成不同狀態。

## 我們需要更清楚資料格式的地方

- 來源類型、查核狀態、回報時間、需求對象、是否第一手資訊需要分開記錄。
- 地點不能只有文字描述；但 Phase 0 不補真實地址或地圖，只標示缺口。
- 服務類型與事件類型應該保留人工覆核紀錄。

## 我們需要人工確認的地方

- 確認資訊是否仍有效。
- 確認來源是否第一手。
- 確認當事人是否同意公開或建立任務。
- 確認缺少的時間、地點與現場條件。
